# **UTCS-ID Specification v1.0**

**Unified Traceability Code Standard — Identifier Format for Lifecycle Artefacts**

**Document ID:** UTCS-SPEC-01  
**Version:** 1.0  
**Status:** RELEASE  
**Owner:** AMPEL360 / OPT-IN Governance  
**Applies to:** All LC Stages, CAOS, GenCCC, Digital Product Passports  
**Replaces:** All previous LC ID / UTCS-ID conventions

---

# **0. Purpose**

The purpose of this specification is to define a **single, unified, extended, deterministic naming and numbering convention** for all artefacts generated throughout the AMPEL360 lifecycle, ensuring:

* Machine readability (CI/CD, GenCCC, CAOS, DPP)
* Human readability (engineering, certification, operations)
* Deterministic derivation from structured taxonomies (ATA, LC stages)
* Long-term traceability, comparability, and auditability
* Compatibility with UTCS (Unique Traceability Code Standard)

This specification enforces a **single formula** for all identifiers:
a unified form that integrates **LC Stage**, **ATA classification**, and **Semantic Functional Role**.

---

# **1. Scope**

This specification applies to:

* All LC artefacts (documents, models, code, datasets, analysis results)
* All UTCS-governed entities (DPP nodes, CAOS operations, GenCCC outputs)
* All repositories under AMPEL360 governance
* Any generated artefact that must appear in:

  * CAOS
  * GenCCC audit graphs
  * V&V Lake
  * Digital Product Passports
  * Engineering and Manufacturing pipelines

It does **not** govern:

* Runtime IDs generated by aircraft systems
* Vendor proprietary numbering schemes
* External certification authority IDs

# **3. Unified UTCS-ID Formula**

AMPEL360 adopts **one single identifier format** for all artefacts:

## **3.1 Identifier Format (canonical + human-readable unified form)**

```
LC-{STAGE}{SUBSTAGE}_{ATAIDX}-{ATADESC}_{FUNC}_VxRy
```

Where every component is **mandatory**.

### **3.2 Components**

#### **3.2.1 LC Stage**

| STAGE | Meaning                                |
| ----- | -------------------------------------- |
| **1** | Development Stage                      |
| **2** | Manufacturing Stage                    |
| **3** | Series Operations (CAOS, MRO, Support) |

#### **3.2.2 Substage Codes**

### Development (Stage 1)

| Code   | Substage                                      |
| ------ | --------------------------------------------- |
| **11** | Research & Concept Development                |
| **12** | Safety Analysis & Preliminary Risk            |
| **13** | Requirements (Material, Process, Functional)  |
| **14** | Design (CAD, Architecture)                    |
| **15** | Interfaces & Constituency Assemblies          |
| **16** | Engineering Analysis, Simulation, Integration |
| **17** | Verification & Validation (V&V)               |

### Manufacturing (Stage 2)

# **4. ATA Index (ATAIDX)**

`ATAIDX` is a **2-, 4-, or 6-digit code** derived from ATA iSpec 2200:

* **2 digits** → System level (e.g., 57 = Wings)
* **4 digits** → Section/Subsection (e.g., 5723 = Slats)
* **6 digits** → Subject/Sub-subsystem (e.g., 572301 = Flap Well Envelope)

---

# **5. Derived ATA Description (ATADESC)**

ATADESC is **automatically generated** using a deterministic ATA dictionary.

### **5.1 Lookup Rules**

Given `ATAIDX`:

1. Extract system code:
   `{sys} = ATAIDX[0:2]`
2. Extract section code if length ≥ 4:
   `{sec} = ATAIDX[0:4]`
3. Extract subject code if length = 6:
   `{subj} = ATAIDX`

### **5.2 Composition Rules**

`ATADESC` is composed as:

```
ATADESC = SYS_DESC [_SECTION_DESC] [_SUBJECT_DESC]
```

Example:

`572301 → WINGS_SLATS_FLAP_WELL_ENV`

### **5.3 Fallback Rule**

If lookup fails at any level:

```
ATADESC = UNKNOWN_ATA_{ATAIDX}
```

# **6. Functional Descriptor (FUNC)**

Short textual code indicating the artefact's functional purpose.

Examples:

| FUNC        | Meaning                          |
| ----------- | -------------------------------- |
| **CFD**     | Computational Fluid Dynamics     |
| **FEM**     | Finite Element Model             |
| **NNMODEL** | Neural Network Model             |
| **VNVREP**  | Verification & Validation Report |
| **MROPROC** | Maintenance Procedure            |
| **CAD**     | CAD geometry                     |
| **INTF**    | Interfaces definition            |

Rules:

* Uppercase
* 3–12 characters
* No spaces
* Must match UTCS functional taxonomy

---

# **7. Versioning Convention**

Mandatory suffix:

```
VxRy
```

Where:

* `x` increments for major design baseline updates (breaking changes)
* `y` increments for revisions (non-breaking updates)

Examples:

* `V1R0` → First release
* `V2R3` → Third revision of second baseline

# **9. Formal Rules**

### **Rule 1 — The unified format is mandatory**

All AMPEL360 artefacts MUST use:

```
LC-{STAGE}{SUBSTAGE}_{ATAIDX}-{ATADESC}_{FUNC}_VxRy
```

### **Rule 2 — Machine compatibility**

Identifiers must be parseable by:

* CAOS
* GenCCC
* CI/CD pipelines
* DPP UTCS resolvers
* V&V Lake

### **Rule 3 — Deterministic derivation**

`ATADESC` MUST be generated automatically from a central ATA catalog.

### **Rule 4 — Uniqueness**

Two artefacts must never share the same UTCS-ID.

### **Rule 5 — Immutability**

Once assigned, UTCS-ID cannot change except for VxRy.

### **Rule 6 — Case**

All descriptors MUST be uppercase.

### **Rule 7 — Delimiters**

* `_` separates structural components
* `-` separates ATAIDX from its descriptor

### **Rule 8 — Length**

No global length limit, but < 200 characters recommended.

# **12. Appendix A — Grammar (EBNF)**

```
utcs-id = "LC-" stage substage "_" ataidx "-" atadesc "_" func "_" version

stage = "1" | "2" | "3"
substage = digit digit

ataidx = 2digit | 4digit | 6digit

atadesc = label { "_" label }
label = 1*ALPHA

func = 3*12(ALPHA | DIGIT)

version = "V" digit "R" digit
```
